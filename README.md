# Quietline

Go + Next.js private messenger MVP built from the high-load WebSocket pub/sub project.

The browser encrypts messages and files before sending them. The Go backend, Postgres, Redis, S3/MinIO, and WebSocket layer only see ciphertext.

## Privacy Model

The server never receives:

- plaintext message text
- room secrets
- browser private identity key

The server does receive:

- public identity data
- room metadata
- encrypted message payloads
- nonces and algorithm metadata
- encrypted file blobs

Current crypto model:

- Browser generates a local ECDH identity keypair.
- Browser generates or accepts a 256-bit room secret.
- Messages are encrypted with AES-GCM using a key derived from the room secret.
- Files are encrypted with AES-GCM in the browser and then uploaded to S3-compatible storage.
- Room secret is stored only in browser `localStorage`.
- Invites are shared out-of-band as `roomId:roomSecret`.
- Room secrets are generated by the browser; the create-room UI does not accept custom secrets.

This is a practical zero-knowledge MVP, not yet a Signal-grade protocol. A production messenger should add X3DH, Double Ratchet, device verification, recovery flows, abuse controls, and durable encrypted key management.

## Architecture

```text
Next.js browser client
  |
  | encrypts plaintext locally
  v
Go REST API: /v1/zk/rooms/{roomId}/messages and /v1/zk/files
  |
  | stores message ciphertext in Postgres, file ciphertext in S3, publishes room event
  v
Redis Pub/Sub
  |
  v
Go WebSocket gateways
  |
  | ciphertext only
  v
Next.js browser clients decrypt locally
```

Each Go instance keeps only local WebSocket connections in memory. Redis fans encrypted room events across instances. Postgres stores identities, room metadata, and encrypted message history. S3/MinIO stores encrypted file blobs.

## Stack

- Backend: Go, standard `net/http`, Gorilla WebSocket
- Realtime broker: Redis Pub/Sub
- Storage: Postgres via `pgx`
- File storage: S3-compatible storage via MinIO
- Frontend: Next.js, React, Mantine, React Query
- UI: responsive mobile/tablet/desktop layout, light/dark themes, RU/EN i18n
- Realtime UX: typing indicators, room presence, last-seen timestamps, chat notifications, system messages, and WebRTC audio-call signaling
- Browser crypto: Web Crypto API, AES-GCM

## Backend API

### Health

```http
GET /healthz
```

### Account Auth

```http
POST /v1/auth/register
```

```json
{
  "username": "alice",
  "password": "correct horse battery staple",
  "displayName": "Alice"
}
```

```http
POST /v1/auth/login
```

```json
{
  "username": "alice",
  "password": "correct horse battery staple"
}
```

Both return:

```json
{
  "accessToken": "eyJ...",
  "tokenType": "Bearer",
  "expiresAt": 1770000000,
  "principal": {
    "clientId": "account-id",
    "userId": "account-id",
    "username": "alice",
    "theme": "dark",
    "scopes": ["publish", "subscribe", "topics:read"],
    "expires": 1770000000
  }
}
```

Protected REST calls use:

```text
Authorization: Bearer <accessToken>
```

Protected WebSocket connections use:

```text
ws://localhost:18080/ws?topics=room:<roomId>&token=<accessToken>
```

### Profile

```http
GET /v1/me
```

Returns the authenticated principal, including the saved UI theme and avatar URL.

```http
PUT /v1/me/theme
```

```json
{
  "theme": "light"
}
```

Allowed values are `light` and `dark`. The frontend applies the selected theme immediately and persists it in the user profile.

```http
PUT /v1/me/avatar
Content-Type: multipart/form-data
```

Upload field: `avatar`.

The frontend crops avatars to a square, resizes them to 256x256, and encodes them as WebP before upload. The backend accepts PNG, JPEG, and WebP avatars up to 1 MiB and stores them in S3/MinIO.

```http
GET /v1/users/{userId}/avatar
```

Returns the stored avatar image.

### Identity

```http
PUT /v1/zk/identities/{userId}
```

```json
{
  "displayName": "Ada",
  "identityPublicKey": "base64url-public-jwk"
}
```

```http
GET /v1/zk/identities/{userId}
```

```http
PUT /v1/zk/identities/{userId}/last-seen
```

Updates and returns the local crypto identity's `lastSeenAt` timestamp.

### Rooms

```http
POST /v1/zk/rooms
```

```json
{
  "name": "Private room",
  "members": ["user-id"]
}
```

```http
GET /v1/zk/rooms?userId=user-id
```

```http
DELETE /v1/zk/rooms/{roomId}/members/{userId}
```

Removes the user from the room membership list. The frontend also drops the local room secret and sends an encrypted leave system message before leaving.

### Encrypted Messages

```http
POST /v1/zk/rooms/{roomId}/messages
```

```json
{
  "senderId": "user-id",
  "ciphertext": "base64url-ciphertext",
  "nonce": "base64url-nonce",
  "algorithm": "AES-GCM-256",
  "keyId": "local-key-id"
}
```

```http
GET /v1/zk/rooms/{roomId}/messages
```

### Encrypted Files

Files are encrypted in the browser before upload.

```http
POST /v1/zk/files
Content-Type: multipart/form-data
Authorization: Bearer <accessToken>
```

Multipart field:

```text
file=<encrypted blob>
```

Response:

```json
{
  "fileId": "object-id",
  "size": 12345
}
```

Download encrypted blob:

```http
GET /v1/zk/files/{fileId}
Authorization: Bearer <accessToken>
```

### WebSocket

Subscribe to encrypted room events:

```text
ws://localhost:8080/ws?topics=room:<roomId>
```

The payload is an event envelope whose `data` field is an encrypted message.

## Using the App

1. Open `http://localhost:3000`.
2. Register an account or login with username/password.
3. Create a local crypto identity. Its private key stays in this browser.
4. Create an encrypted room.
5. Copy the invite. It contains `roomId:roomSecret`; room ids are masked in the UI and can be copied with the clipboard button.
6. Open another browser/profile, login/register, create a local crypto identity, and paste the invite into `Import invite`.
7. Upload an avatar from the profile card if needed. The browser crops and compresses it before upload.
8. Toggle light/dark theme from the top bar if needed. The choice is saved in the account profile.
9. Switch RU/EN language from the top bar if needed. The choice is saved locally in the browser.
10. On mobile, use the top tabs to move between Chat, Rooms, and Profile; selecting a room opens the chat immediately.
11. Send messages and attach files. Backend, Redis, Postgres, and MinIO see ciphertext only.
12. Download attachments from the download icon in each file card. The browser decrypts the file before saving it.
13. Open a participant profile from a member badge or message author to inspect display name, status, last seen time, public key, and avatar.
14. Leave a room from the chat header if you want to remove your membership and local room secret.
15. Typing indicators, online presence, browser notifications, and encrypted system messages are delivered over WebSocket.
16. `lastSeenAt` is stored with the local crypto identity. Users stay online while the room WebSocket tab is open and move offline when the tab closes or they log out.
17. Audio calls are WebRTC peer-to-peer. The backend only relays signaling events through the room WebSocket topic.

## Auth

The Go backend supports two auth flows:

- `POST /v1/auth/register`
- `POST /v1/auth/login`
- `POST /v1/auth/token`
- Bearer token validation
- `AUTH_ENABLED=true`

Docker Compose uses `AUTH_ENABLED=true`. Passwords are stored as bcrypt hashes in Postgres.

`POST /v1/auth/token` is kept for API-key style service clients. The frontend uses username/password registration and login.

Account identity and crypto identity are intentionally separate in this MVP:

- account identity controls access to the API
- account profile stores UI preferences, including light/dark theme and avatar metadata
- crypto identity and room secrets control message confidentiality

## Deployment Notes

Recommended production shape:

```text
TLS Load Balancer
  |
  +-- Next.js frontend
  |
  +-- Go WebSocket/API replicas
        |
        +-- Redis
        +-- Postgres
        +-- S3-compatible object storage
```

Use TLS everywhere, managed Postgres, managed Redis, health probes on `/healthz`, Prometheus scrape on `/metrics`, strong secrets from a secret manager, and explicit CORS origins.

# Quietline

Go + Next.js private messenger MVP built from the high-load WebSocket pub/sub project.

The browser encrypts messages and files before sending them. The Go backend, Postgres, Redis, S3/MinIO, and WebSocket layer only see ciphertext.

## Encryption Model

Messages and files are encrypted in the browser before sending. The server stores only ciphertext.

What the server stores:

- encrypted message payloads (ciphertext + nonce + algorithm)
- room secrets (generated by the browser, persisted in Postgres)
- encrypted file blobs in S3/MinIO
- public identity data (display name, public key string)
- room and membership metadata

What the server never has:

- plaintext message content
- private keys

Current crypto model:

- Browser generates a 256-bit room secret when creating a room.
- Messages are encrypted with AES-GCM-256 using a key derived from the room secret.
- Files are encrypted with AES-GCM in the browser before upload.
- Room secret is stored in Postgres (server-side) and in browser `localStorage`.
- Invites are shared out-of-band as opaque `ql_...` invite tokens. The token contains the room id and room secret for compatibility, but the UI no longer displays those internals directly.
- Each user has one identity per account. The identity stores a display name and an optional public key string; there is no per-device ECDH keypair rotation.

This is a client-encrypted messenger MVP, not a Signal-grade protocol. The server cannot read message content but does hold room secrets. A production-grade system should add X3DH, Double Ratchet, device verification, recovery flows, and durable encrypted key management.

## Architecture

```text
Next.js browser
  |
  | encrypts plaintext with AES-GCM (room-derived key)
  v
Go REST API  /v1/chat/rooms/{roomId}/messages  /v1/chat/files
  |
  | stores ciphertext + nonce in Postgres
  | stores room metadata and room secret in Postgres
  | stores encrypted file blobs in S3/MinIO
  | publishes room event
  v
Redis Pub/Sub
  |
  v
Go WebSocket gateways  (forward ciphertext-only events)
  |
  v
Next.js browser clients decrypt locally
```

Each Go instance keeps only local WebSocket connections in memory. Redis fans room events across instances. Postgres stores identities, rooms, room secrets, and encrypted message history. S3/MinIO stores encrypted file blobs.

API paths `/v1/zk/`, `/v1/chat/`, and `/v1/chats/` are equivalent aliases — all route to the same handlers.

## Stack

- Backend: Go 1.25, standard `net/http`, Gorilla WebSocket 1.5
- Realtime broker: Redis 7.4 Pub/Sub
- Storage: PostgreSQL 17 via `pgx/v5`
- File storage: S3-compatible storage via MinIO (minio-go/v7)
- Frontend: Next.js 15 (App Router), React 18, Mantine 7, React Query 5
- UI: responsive mobile/tablet/desktop layout, light/dark themes, RU/EN i18n
- Realtime UX: typing indicators, room presence, last-seen timestamps, chat notifications, system messages, and WebRTC audio-call signaling
- Browser crypto: Web Crypto API, AES-GCM

## Backend API

### Health

```http
GET /healthz
```

```http
GET /metrics
```

Returns Prometheus metrics (active connections, message counters, latency histograms).

> **API path aliases**: All `/v1/zk/` endpoints are also reachable as `/v1/chat/` and `/v1/chats/`. The docs below use the canonical `/v1/zk/` prefix.

### Account Auth

```http
POST /v1/auth/register
```

```json
{
  "username": "alice",
  "password": "correct horse battery staple",
  "displayName": "Alice"
}
```

```http
POST /v1/auth/login
```

```json
{
  "username": "alice",
  "password": "correct horse battery staple"
}
```

Both return:

```json
{
  "accessToken": "eyJ...",
  "tokenType": "Bearer",
  "expiresAt": 1770000000,
  "principal": {
    "clientId": "account-id",
    "userId": "account-id",
    "username": "alice",
    "theme": "dark",
    "scopes": ["publish", "subscribe", "topics:read"],
    "expires": 1770000000
  }
}
```

Protected REST calls use:

```text
Authorization: Bearer <accessToken>
```

Protected WebSocket connections use:

```text
ws://localhost:18080/ws?topics=room:<roomId>&token=<accessToken>
```

### Profile

```http
GET /v1/me
```

Returns the authenticated principal, including the saved UI theme and avatar URL.

```http
PUT /v1/me/theme
```

```json
{
  "theme": "light"
}
```

Allowed values are `light` and `dark`. The frontend applies the selected theme immediately and persists it in the user profile.

```http
PUT /v1/me/avatar
Content-Type: multipart/form-data
```

Upload field: `avatar`.

The frontend crops avatars to a square, resizes them to 256x256, and encodes them as WebP before upload. The backend accepts PNG, JPEG, and WebP avatars up to 1 MiB and stores them in S3/MinIO.

```http
GET /v1/users/{userId}/avatar
```

Returns the stored avatar image.

```http
GET /v1/me/identity
```

Returns the authenticated user's own crypto identity (shortcut to `GET /v1/zk/identities/{userId}`).

### Two-Factor Authentication (TOTP)

```http
POST /v1/me/2fa/setup
```

Generates a TOTP secret and returns a `otpauth://` URI (for QR display) along with the raw secret. The secret is not yet active until confirmed.

```http
POST /v1/me/2fa/confirm
```

```json
{ "code": "123456" }
```

Confirms the TOTP code from the authenticator app. Activates 2FA on the account.

```http
DELETE /v1/me/2fa
```

```json
{ "code": "123456" }
```

Disables 2FA. Requires a valid TOTP code.

### Sessions

```http
GET /v1/me/sessions
```

```http

```

Revokes all sessions except the current one.

```http

```


```text

```

## Using the App

1. Open `http://localhost:3000`.
2. Register an account or login with username/password.
3. Create an encrypted room.
4. Copy the invite. It is shown as an opaque `ql_...` token or invite link, without exposing the room id or secret directly in the UI.
5. Open another browser/profile, login/register, and paste the invite into `Import invite`.
6. Upload an avatar from the profile card if needed. The browser crops and compresses it before upload.
7. Toggle light/dark theme from the top bar if needed. The choice is saved in the account profile.
8. Switch RU/EN language from the top bar if needed. The choice is saved locally in the browser.
9. On mobile, use the top tabs to move between Chat, Rooms, and Profile; selecting a room opens the chat immediately.
10. Send messages and attach files. The backend stores only ciphertext; Postgres and MinIO never see plaintext.
11. Download attachments from the download icon in each file card. The browser decrypts the file before saving it.
12. Open a participant profile from a member badge or message author to inspect display name, status, last seen time, public key, and avatar.
13. Leave a room from the chat header to remove your membership.
14. Typing indicators, online presence, browser notifications, and encrypted system messages are delivered over WebSocket.
15. `lastSeenAt` is stored on the identity record. Users stay online while the room WebSocket tab is open and move offline when the tab closes or they log out.
16. Audio calls are WebRTC peer-to-peer. The backend only relays signaling events through the room WebSocket topic.

## Auth

The Go backend supports two auth flows:

- `POST /v1/auth/register`
- `POST /v1/auth/login`
- `POST /v1/auth/token`
- Bearer token validation
- `AUTH_ENABLED=true`

Docker Compose uses `AUTH_ENABLED=true`. Passwords are stored as bcrypt hashes in Postgres.

`POST /v1/auth/token` is kept for API-key style service clients. The frontend uses username/password registration and login.

Each user has a single identity that covers both account access and messaging:

- account identity controls access to the API (scopes: `publish`, `subscribe`, `topics:read`)
- account profile stores display name, avatar, theme preference
- the same user ID is used for room membership, message authorship, and identity lookups
- an optional `identityPublicKey` field on the identity can store a user-supplied public key string for future upgrade to per-user asymmetric encryption

## Deployment Notes

Recommended production shape:

```text
TLS Load Balancer
  |
  +-- Next.js frontend
  |
  +-- Go WebSocket/API replicas
        |
        +-- Redis
        +-- Postgres
        +-- S3-compatible object storage
```

Use TLS everywhere, managed Postgres, managed Redis, health probes on `/healthz`, Prometheus scrape on `/metrics`, strong secrets from a secret manager, and explicit CORS origins.
